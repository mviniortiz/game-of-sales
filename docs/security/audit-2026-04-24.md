## C-01 · `demo_requests` sem RLS ativada — leak de leads

**Severidade:** 🔴 Critical
**Impacto:** LGPD violation · Vazamento de PII de prospects · Potencial multa até R$50M
**Confiança:** Confirmado via query

### Descoberta

A tabela `public.demo_requests` tem políticas RLS definidas (`service_role_all_demo_requests`, `authenticated_update_demo_requests`) mas **RLS está DISABLED** no nível da tabela. Isso significa que as policies são ignoradas e qualquer usuário autenticado no projeto Supabase consegue fazer `SELECT *` na tabela.

### Dados expostos

```
id, name, email, company, phone, source, status, calendly_event_uri,
notes, created_at, updated_at, utm_source, utm_medium, utm_campaign,
utm_term, utm_content, gclid, fbclid, referrer, landing_page
```

São **dados pessoais LGPD sensíveis** de cada prospect. Combinados, permitem identificar indivíduos e reconstruir campanhas de mídia.

### POC

```sql
-- Com qualquer JWT autenticado:
SELECT name, email, phone FROM demo_requests LIMIT 100;
-- Retorna dados de todos leads, ignorando policies.
```

### Fix

```sql
ALTER TABLE public.demo_requests ENABLE ROW LEVEL SECURITY;

-- Verificar policies existentes são adequadas:
-- service_role_all_demo_requests: OK (bypass RLS)
-- authenticated_update_demo_requests: precisa revisar se permite só admins
--
-- Adicionar policy SELECT restritiva (super_admin + admins da company dona)
DROP POLICY IF EXISTS demo_requests_select_admins ON public.demo_requests;
CREATE POLICY demo_requests_select_admins ON public.demo_requests
    FOR SELECT
    USING (
      EXISTS (SELECT 1 FROM user_roles WHERE user_id = auth.uid() AND role = 'admin')
      OR EXISTS (SELECT 1 FROM profiles WHERE id = auth.uid() AND is_super_admin = true)
    );
```

## C-02 · `user_roles` sem RLS — escalação de privilégio

**Severidade:** 🔴 Critical
**Impacto:** Qualquer usuário pode virar admin
**Confiança:** Confirmado

### Descoberta

Tabela `public.user_roles` tem policy `user_roles_all` mas **RLS está DISABLED**. Qualquer usuário autenticado pode inserir na tabela:

```sql
INSERT INTO user_roles (user_id, role) VALUES (auth.uid(), 'admin');
```

E virar admin do sistema, acessando `AdminVendedores`, `AdminCompanies`, etc.

### Fix

```sql
ALTER TABLE public.user_roles ENABLE ROW LEVEL SECURITY;

-- Policy: só super_admins podem escrever
DROP POLICY IF EXISTS user_roles_all ON public.user_roles;
CREATE POLICY user_roles_select_own ON public.user_roles
    FOR SELECT
    USING (user_id = auth.uid() OR
           EXISTS (SELECT 1 FROM profiles WHERE id = auth.uid() AND is_super_admin = true));

CREATE POLICY user_roles_admin_write ON public.user_roles
    FOR ALL
    USING (EXISTS (SELECT 1 FROM profiles WHERE id = auth.uid() AND is_super_admin = true));
```

## C-03 · `eva-stale-deal-followup` sem verify_jwt

**Severidade:** 🔴 Critical
**Impacto:** Drain de créditos LLM · Abuse potencial · Custo financeiro direto
**Confiança:** Confirmado (POC executada)

### Descoberta

A edge function `eva-stale-deal-followup` foi deployada com flag `--no-verify-jwt`. Qualquer um na internet consegue invocar sem autenticação:

### POC

```bash
curl -X POST https://omsdkjzkphflpwnbaeye.supabase.co/functions/v1/eva-stale-deal-followup \
  -H "Content-Type: application/json" \
  -d '{"force": true}'

# Resposta (sem token nenhum):
# {"ok":true,"scanned":1,"processed":0,"skipped_existing":1,"failed_llm":0,"errors":[]}
```

Cada chamada consome tokens do Anthropic Claude (ou fallback OpenAI) por deal parado. Um atacante pode fazer milhares de calls/min drenando o saldo.

### Fix imediato

```bash
# Re-deploy com verify_jwt
npx supabase functions deploy eva-stale-deal-followup
# (sem --no-verify-jwt)
```

Ou adicionar auth signature interna:

```typescript
// No início do handler
const INTERNAL_SECRET = Deno.env.get("EVA_CRON_SECRET")!;
if (req.headers.get("x-cron-secret") !== INTERNAL_SECRET) {
  return new Response(JSON.stringify({ error: "unauthorized" }), { status: 401 });
}
```

Migration do pg_cron precisa passar esse header também.

## C-04 · JWT anon_key hardcoded em migration commitada

**Severidade:** 🔴 Critical (hygiene) / 🟡 Medium (impact real)
**Impacto:** Anon key vaza no repositório Git público/privado
**Confiança:** Confirmado

### Descoberta

Arquivo `supabase/migrations/20260416_evolution_keepwarm_cron.sql` tem:

```sql
'Authorization', 'Bearer eyJhbGciOiJIUzI1NiIsInR5cCI6IkpXVCJ9.eyJpc3MiOiJzdXBhYmFzZSIs...'
```

Esse é o JWT anon_key do projeto, expiração 2036. Embora anon_key seja "pública" (usada no frontend SPA), commitar em migration é:

1. Permanente no git history (mesmo que delete depois)
2. Sincronizado pro GitHub
3. Se repo for público ou um colaborador vazar, scanners de bots pegam

### Fix

1. **Remover o JWT da migration** (substituir por `current_setting('app.settings.anon_key')` ou secret em vault)
2. **Rotacionar o anon_key** se quiser ser cuidadoso (raro, mas possível via Supabase Dashboard)
3. **Adicionar pre-commit hook** ou action que grepa `eyJhbGciOiJIUzI1NiI` e `sk-ant` e falha o push

## H-02 · SPF ausente + DMARC policy=none — email spoofing

**Severidade:** 🟠 High
**Impacto:** Atacantes enviam emails fingindo ser do domínio vyzon.com.br (phishing)

### Descoberta

- TXT record do domínio: só `google-site-verification=...`, sem SPF
- DMARC: `v=DMARC1; p=none;` (policy none = não rejeita spoofed)

Qualquer pessoa pode enviar email com `From: ceo@vyzon.com.br` ou `financeiro@vyzon.com.br` e o email chega na caixa do destinatário sem rejeição.

### Fix

Adicionar via DNS (provider registro.br):

```
vyzon.com.br. IN TXT "v=spf1 include:amazonses.com include:_spf.google.com ~all"
_dmarc.vyzon.com.br. IN TXT "v=DMARC1; p=quarantine; rua=mailto:dmarc@vyzon.com.br; ruf=mailto:dmarc@vyzon.com.br; fo=1; aspf=r; adkim=r;"
```

Em 1-2 semanas, subir DMARC para `p=reject`.

DKIM precisa configurar também pro AWS SES (gerar par de chaves pela AWS, publicar public key como TXT).

## H-03 · Content Security Policy (CSP) ausente

**Severidade:** 🟠 High
**Impacto:** Ausência de mitigação contra XSS injection

### Descoberta

```bash
curl -sI https://vyzon.com.br/ | grep -i content-security-policy
# (nenhum output — CSP ausente)
```

Se um atacante conseguir injetar `<script>` via qualquer caminho (XSS stored no CRM, por exemplo em nome de deal), sem CSP o script roda livremente.

### Fix

Adicionar header no `vercel.json`:

```json
{
  "headers": [
    {
      "source": "/(.*)",
      "headers": [
        {
          "key": "Content-Security-Policy",
          "value": "default-src 'self'; script-src 'self' 'unsafe-inline' https://www.googletagmanager.com https://connect.facebook.net https://www.clarity.ms; style-src 'self' 'unsafe-inline' https://fonts.googleapis.com; font-src 'self' https://fonts.gstatic.com data:; img-src 'self' data: https: blob:; connect-src 'self' https://*.supabase.co https://www.google-analytics.com https://connect.facebook.net https://www.clarity.ms wss://*.supabase.co; frame-src 'self' https://www.googletagmanager.com; object-src 'none'; base-uri 'self'; form-action 'self'; frame-ancestors 'none'; upgrade-insecure-requests"
        }
      ]
    }
  ]
}
```

Testar em staging primeiro — CSP pode quebrar scripts inline não previstos. Usar modo `Content-Security-Policy-Report-Only` por 1-2 semanas antes de ativar enforcement.

## H-05 · CORS wildcard em edge functions públicas

**Severidade:** 🟠 High (contextual)
**Impacto:** Permite fetch cross-origin de origens não autorizadas

### Descoberta

Todas edge functions retornam `Access-Control-Allow-Origin: *`. Pra webhooks é OK, mas pra functions que lidam com dados sensíveis (como `upload-ads-conversion` que recebe lead_id) deveria restringir.

### Fix

```typescript
const ALLOWED_ORIGINS = [
  "https://vyzon.com.br",
  "https://www.vyzon.com.br",
  "http://localhost:5173", // dev
];

const origin = req.headers.get("origin") || "";
const allowOrigin = ALLOWED_ORIGINS.includes(origin) ? origin : ALLOWED_ORIGINS[0];

const corsHeaders = {
  "Access-Control-Allow-Origin": allowOrigin,
  "Access-Control-Allow-Headers": "authorization, x-client-info, apikey, content-type",
  "Access-Control-Allow-Methods": "POST, OPTIONS",
  "Access-Control-Allow-Credentials": "true",
};
```

## H-06 · Ausência de rate limit em edge functions públicas

**Severidade:** 🟠 High
**Impacto:** DoS + drain de recursos computacionais/LLM

### Descoberta

```bash
for i in {1..10}; do curl -s -o /dev/null -w "%{http_code} " -X POST https://.../functions/v1/lead-webhook; done
# 400 400 400 400 400 400 400 400 400 400
```

10 requests consecutivos passaram sem 429. A função `lead-webhook` não tem rate limit, e roda lookup em `lead_webhooks` table por chamada.

### Fix

Usar a função `consume_rate_limit` que já existe no schema:

```typescript
const { data, error } = await adminClient.rpc("consume_rate_limit", {
  p_bucket: `lead_webhook:${req.headers.get("cf-connecting-ip") || "unknown"}`,
  p_limit: 60,
  p_window_seconds: 60,
});
if (!data?.allowed) {
  return new Response(JSON.stringify({ error: "rate_limited" }), { status: 429 });
}
```

Adicionar em todas edge functions públicas (webhooks, eva-stale-deal-followup após fix do C-03, calendar-slots).

## M-01 · Trigger `update_deals_updated_at` atualiza em cada UPDATE

**Severidade:** 🟡 Medium (DevX) / 🔵 Info (Security)
**Impacto:** Complicações em operações manuais; não é security em si

O trigger força `updated_at = now()` em cada UPDATE em `deals`. Isso dificulta testes que precisam simular staleness. Não é vulnerabilidade — é observação.

### Fix (opcional)

Permitir override via SQL se `updated_at` for explicitamente setado:

```sql
CREATE OR REPLACE FUNCTION update_deals_updated_at()
RETURNS trigger AS $$
BEGIN
  IF NEW.updated_at IS NOT DISTINCT FROM OLD.updated_at THEN
    NEW.updated_at = now();
  END IF;
  RETURN NEW;
END;
$$ LANGUAGE plpgsql;
```

## M-03 · URLs de edge functions expostas em client bundles
